The primary driver for this was my ongoing frustration with what I'll call "scope creep". This is the drifting off target that occurs as models hit errors, find fixes, develop workarounds, and generally do whatever they can to find an answer. Inevitably we've all found ourselves telling models and agents "stop" and more often than not, this is a result of "scope creep".

Before we go any further, it's important to acknowledge [GSD](https://github.com/gsd-build/get-shit-done) (GET SHIT DONE), a project that describes itself as "A light-weight and powerful meta-prompting, context engineering and spec-driven development system for Claude Code by TÂCHES."

GSD currently utilizes projects.md and has thousands of happy users who attest to it's efficacy. This project also draws from the now infamous https://github.com/obra/superpowers repository.

## THE CORE IDEA

One file. One binary. The file is the project. The binary manages the file.

## THE FILE: project.md

Every project gets a single markdown file that contains everything:

- What we're building (What This Is, Core Value)

- What we need (Requirements — three tiers)

- Where we are (Current State — the resume point)

- What we've decided (Key Decisions with outcomes)

- What to do (Tasks — phase-grouped checklist)

- What we've learned (Discoveries, Session Log)

## THE LIFECYCLE

DEFINE → DESIGN → BUILD → VERIFY → SHIP

Each phase has its own section in the Tasks. You move through them sequentially. The projectsmd phase transition command handles the handoff — it runs an Evolution checklist that promotes requirements, updates decisions, and checks if the project description still holds.

This is where the magic happens. Every work session follows the same pattern:

The key insight: Current State is the resume point. It says exactly what phase you're in, what was last done, and what to do next. An agent (or human) reading ONLY that section can pick up immediately.

## THE REQUIREMENTS LIFECYCLE

Requirements aren't just "done" or "not done." They move through tiers: This prevents silent scope creep. If something didn't work, it moves to Out of Scope with a reason — so you don't re-try it later.

## THE DECISION TRACKING

Every decision gets logged with:

- What was decided

- Why (rationale)

- Whether it worked (outcome: ✓ Good / ⚠️ Revisit / — Pending)

Over time, you build a decision history that shows what worked and what didn't. The #1 context lost between sessions is "why did we choose X over Y?" — this captures it.

## THE AGENT INTEGRATION

This is what makes it work for AI agents:

1. Install the skill: projectsmd skill install

2. Agent discovers the skill (progressive disclosure)

3. Agent reads project.md at session start

4. Agent uses CLI commands during work

5. Agent runs projectsmd session at end

The skill teaches agents the exact commands to use at each step.

No framework-specific code — it works with Hermes Agent, Claude Code, Cursor, Codex, or any agent that supports the agentskills.io standard.

## THE KEY DIFFERENCE

Traditional: context lives in your head, Slack, Jira, Google Docs. When the session ends, it evaporates. ProjectsMD: context lives in one file. When the session ends, it's preserved. When the next session starts, it's restored. The file IS the project.
